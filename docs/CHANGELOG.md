### Decisions and Failures

- Why the exception channel was rejected for `run_fixer_script`: raising on non-zero exit
  inside a module-scoped autouse fixture errors every parametrized test in the module, not just
  the one file that triggered the fix. Per-file fixer outcomes (clean, needs-fix, fixed,
  unexpected error) belong in per-file report data -- the fixture should collect them and the
  per-file test should assert on them. Using exceptions conflates environment failure (broken
  interpreter, missing script -- genuine RuntimeError territory) with normal fixer output
  (non-zero exit is a documented, meaningful result). The return-tuple contract makes the
  distinction explicit and puts outcome handling in the right place.
- Root cause of the reset_repo initial-setup no-op: regression introduced when `reset_repo.py`
  switched from shelling out to `propagate_style_guides.py` to calling `repolib` directly; the
  batch-mode self-skip guard was correct for batch propagation but silently disabled the entire
  initial-setup use case. E2E in `/tmp` scratch clones confirmed all 18 typescript files land
  and the TYPESCRIPT gitignore block is written. Full live exit-0 run requires a committed HEAD
  (uncommitted-overlay clones cause `git rm` to refuse modified files -- test-setup artifact,
  not a product bug).

### Decisions and Failures

- **Deferred 8 optional flags after audit.** The CLI surface started at 9 candidate optional flags (`--format`, `--margin`, `--media`, `--viewport-width`, `--viewport-height`, `--wait-until`, `--no-background`, `--scale`) plus headers/footers and multi-input merge; all were cut to ship only `--landscape`. Rationale: base utilities should serve concrete use cases, not preemptive "someone might want to...". The three v1 use cases (reports, simple docs, CI artifacts) all fit within hardcoded Letter/0.6in/screen/1440x1200. Flags will be added (and documented in PLAYWRIGHT_USAGE.md) only when a real consumer delivers a use case that demands them. This decision mirrors the minimalism principle from docs/PYTHON_STYLE.md ARGPARSE section.
- **Corrected npm set-script form in doc and test design.** Initial doc example used the legacy `npm set-script pdf ...` form; npm v9 removed `set-script` in favor of `npm pkg set`. Corrected the new PLAYWRIGHT_USAGE.md section to recommend `npm pkg set` if consumers need to add the script manually, bringing the docs into alignment with current npm tooling.
- **Corrected test case for relative-path input.** Initial test design passed absolute tmpdir paths in all cases; revised to write the HTML file to REPO_ROOT (via a subdirectory inside the repo) and pass a relative path from REPO_ROOT, ensuring the test exercises the documented `node:path` `resolve()` code path and does not mask resolution bugs that would only surface when users invoke the tool from the repo root.
- **Rejected callback-dict abstraction for buckets (WP-H Patch 9 / WP-E1 acceptance).** Considered routing the four bucket loops through a `{bucket_name: handler_callable}` dict to remove per-bucket inline branching. Rejected in favor of explicit per-bucket inline branches inside `apply_file_bucket()` because each bucket has different policy (noexist skip-if-present, devel new-file handling, test auto-discovery interaction) that reads better in place than spread across four small handler functions and a registration dict. Matches the WP-E1 acceptance criteria.
- **Dropped `prompt_for_type` extraction (Phase A WP-C1 decision, restated for Phase B closure).** A standalone `prompt_for_type()` helper was on the Phase A list but was dropped after `read_repo_type()` already centralized prompting through `parse_repo_type_choice()`. Adding a second helper would have re-split logic that now reads in a single function.
- **No `--verbose` CLI flag for OUTPUT SUPPRESSION.** Quiet-by-default suppression of `NO CHANGE` and `SKIP` lines stays unconditional; detail is preserved in the per-repo and global summary counters. A `--verbose` flag aimed at re-enabling line-level no-op output was considered and deferred because no real consumer has asked for it. Revisit only when a real use case appears. Boundary clarification: the subsequently-added `-v` / `--verbose-paths` flag (see Additions entry above) is a distinct concern -- it toggles PATH DISPLAY (repo-relative vs full absolute paths in action lines) and does NOT re-enable suppressed `NO CHANGE` / `SKIP` lines. The two flags address different problems and do not contradict.
