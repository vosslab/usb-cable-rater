# Closeout report 2026-06-15

This is the final status and known-limitations report for usb-cable-rater, a
macOS Swift CLI that detects USB-C cables live and reports a speed bucket. It
records what works, the macOS-bounded limits that are not code defects, and the
realistic future direction.

## Working

- Live watch detects every port macOS reports active across all three ports when
  a CC-terminating partner is attached.
- `--once` and the watch-startup path produce parity (the same verdict for the
  same physical state).
- Distinct unplug and replug rendering: an unplug is shown as its own event, not
  confused with a re-detect.
- Two-line, colored, honest per-port output: a colored bucket plus an evidence
  detail line, with wording that does not overclaim.
- E-marker decode pipeline proven for the 10G, 40G, and 80G buckets when an SOP'
  node appears (fixture-driven proof).
- M5 device-speed fallback: when a far-end device enumerates, the tool reports an
  observed `At least <speed> [device]` floor from the negotiated link.
- 220 tests pass.
- Two HIGH crash guards fixed (IONotificationPortCreate nil-check; baseAddress
  force-unwrap).

## Known limitations (macOS-bounded, not code)

- macOS exposes the cable e-marker (SOP') only when it has run Discover Identity,
  which it does NOT do merely because power is present.
- A direct 60W+ charger surfaced `AC Charger Connected` with NO device connected
  and NO SOP' node on the tested M1: the power path works, but the e-marker is
  not exposed. "Power present" does not imply "e-marker visible".
- The "Up to 40 Gb/s" speed in System Information is the Mac receptacle and bus
  CAPABILITY, not the attached cable's rating. It is not cable evidence.
- User-space tools (this one and whatcable) can only READ what macOS has already
  queried; they cannot force Discover Identity.
- A bare open-ended cable is electrically invisible: no CC termination, so no CC
  attach and no signal to read.
- A power-only partner (a charger, or a hub whose USB data side does not
  enumerate) gives presence, not a rating.

## Future direction

- Rate an unmarked cable via a far-end USB3+ device that actually enumerates
  (SSD or display), reading the observed `At least <speed>` floor as an observed
  capability rather than a proven cable maximum.
- Address the backend audit's open items: M1 occSource mislabel on PD-identity
  admission, D1 hardcoded usbCPortType for non-M1 SoCs, the incomplete JSON
  control-char escaping (full U+0000 through U+001F range), and the coverage gaps
  (debounce-window silent drop, location-based portNumber recovery, SOP''
  headline fallback).
